# スクラムボード 要求仕様書

スクラム開発におけるタスクを視覚的に管理するカンバンボードアプリケーション。チームメンバーはタスクの作成・編集・ステータス変更・優先度設定・担当者割り当てを行い、スプリントの進捗をリアルタイムで把握できる。データは Supabase（クラウドDB）に保存され、どのデバイスからでもアクセス可能。プロジェクト単位でタスク・スプリントを分離し、ヘッダーのセレクターで切り替えて運用できる。固定ユーザーID・パスワードによるログイン認証でアクセスを制限できる。プロジェクトごとに担当者リストを管理し、タスク追加時のデフォルト担当者を設定できる。

**用語定義**

| 用語 | 定義 |
|------|------|
| タスク | 作業単位。タイトル・説明・優先度・担当者・ストーリーポイントを持つ |
| カラム | タスクのステータスを表す列。プロダクトバックログ / 未着手 / 進行中 / レビュー / 完了 の5種 |
| ボード | 全カラムを並べた作業画面 |
| 優先度 | タスクの重要度。高・中・低 の3段階 |
| 担当者 | タスクを担当するメンバーの名前 |
| プロジェクト | タスク・スプリントをまとめる管理単位。名前・IDを持ち、Supabase の `projects` テーブルで管理する |
| スプリント | 一定期間（通常1〜4週間）の開発イテレーション。開始日・終了日・名称を持つ |
| ストーリーポイント | タスクの相対的な作業量見積もり値。整数で表す |
| ベロシティ | スプリントで完了したタスクのストーリーポイント合計値 |
| バーンダウンチャート | スプリント期間中の残ストーリーポイントの推移を示すグラフ |

**対象外（将来対応）**

- ユーザーごとのデータ分離（ログインユーザーに紐づく Row Level Security）
- タスクへのラベル・コメント機能
- スプリントごとのチームキャパシティ管理

## ＜タスク管理＞

### R-001 タスク作成 #タスク #登録

> **理由：** 作業項目をシステムに登録できなければボードとして機能しないため

###### ＜タスク作成仕様＞

- □□□ S-001-01 各カラムの下部に「+ タスクを追加」ボタンを表示する
- □□□ S-001-02 タスク追加フォームはタイトル（必須）・説明（任意）・優先度・担当者の入力欄を持つ
- □□□ S-001-03 タイトルが空の場合、追加ボタンを非活性にする
- □□□ S-001-04 追加完了またはキャンセル後、フォームを閉じてボタン表示に戻す
- □□□ S-001-05 優先度の初期値は「中」とする
- □□□ S-001-06 タスク追加フォームにストーリーポイントの入力欄（任意・整数）を持つ

### R-002 タスク編集 #タスク #編集

> **理由：** 登録後に内容の変更が生じた場合に対応できるようにするため

###### ＜タスク編集仕様＞

- □□□ S-002-01 カード右上の「⋯」ボタンをクリックするとモーダルを表示する
- □□□ S-002-02 モーダルではタイトル・説明・優先度・担当者をすべて編集できる
- □□□ S-002-03 タイトルが空の場合、保存ボタンを非活性にする
- □□□ S-002-04 Escキー押下またはモーダル背景クリックでモーダルを閉じる
- □□□ S-002-05 モーダルを開いた際、タイトル入力欄にフォーカスを当てる
- □□□ S-002-06 モーダルでストーリーポイントを編集できる

### R-003 タスク削除 #タスク #削除

> **理由：** 不要になったタスクをボードから除去できるようにするため

> **説明：** 削除はカード上に直接ボタンを設けず、編集モーダル内からのみ行う。削除ボタン押下後に確認メッセージを表示する二段階確認とし、誤削除を防ぐ。

###### ＜タスク削除仕様＞

- □□□ S-003-01 削除はタスク編集モーダル内の「削除」ボタンから行う
- □□□ S-003-02 削除ボタン押下時は確認メッセージを表示し、再度確認を求める
- □□□ S-003-03 確認後に削除を実行し、モーダルを閉じる

### R-007 優先度表示 #優先度 #表示

> **理由：** タスクの重要度を一目で把握し、対応順序の判断を支援するため

###### ＜優先度表示仕様＞

- □□□ S-007-01 タスクカードに優先度バッジ（高・中・低）を表示する
- □□□ S-007-02 優先度ごとに色を変える（高：赤、中：黄、低：緑）

### R-008 担当者表示 #担当者 #表示

> **理由：** 誰がどのタスクを担当しているかをボード上で把握できるようにするため

###### ＜担当者表示仕様＞

- □□□ S-008-01 タスクに担当者名を設定できる（任意項目）
- □□□ S-008-02 担当者が設定されている場合、カード下部に名前の頭文字アバターと名前を表示する
- □□□ S-008-03 アバターの背景色は担当者名から一意に決定する

### R-012 ストーリーポイント #ストーリーポイント #見積もり

> **理由：** タスクの作業量を数値化し、スプリント計画とベロシティ計測の基礎データとするため

###### ＜ストーリーポイント仕様＞

- □□□ S-012-01 タスクにストーリーポイント（正の整数）を設定できる（任意）
- □□□ S-012-02 ストーリーポイントが設定されたカードには値をバッジ表示する
- □□□ S-012-03 各カラムのヘッダーにそのカラム内タスクのストーリーポイント合計を表示する
- □□□ S-012-04 スプリントのストーリーポイント合計（計画値）をスプリント情報として表示する
- □□□ S-012-05 カード上のストーリーポイントバッジをクリックすると数値を直接編集できる
- □□□ S-012-06 ストーリーポイント未設定のカードにも「– pt」の編集可能バッジを表示する

### R-016 タスク期限管理 #期限 #タスク

> **理由：** タスクごとに期限を設定し、遅延リスクを視覚的に把握できるようにするため

> **説明：** 完了カラムのタスクは期限超過でも色変化しない。期限の色分け基準は超過：赤、2日以内：橙、設定済み：紫とする。

###### ＜期限管理仕様＞

- □□□ S-016-01 タスクカードに期限バッジ（📅 MM/DD 形式）を表示する
- □□□ S-016-02 期限バッジをクリックすると日付入力フィールドに切り替わりインライン編集できる
- □□□ S-016-03 期限超過のタスクは赤色、2日以内は橙色、設定済みは紫色でバッジを表示する
- □□□ S-016-04 完了カラムのタスクは期限超過でも色変化しない
- □□□ S-016-05 タスク編集モーダルからも期限を設定・変更できる
- □□□ S-016-06 期限は Supabase の `due_date` カラム（text 型、YYYY-MM-DD）に保存する

## ＜ボード表示＞

### R-004 カンバンボード表示 #ボード #表示

> **理由：** タスクの全体像とステータス分布を一覧で把握できるようにするため

> **説明：** プロダクトバックログカラムは `sprintId = null` のタスクのみ表示する。スプリント割り当て済みのタスクはアクティブスプリントのボードにのみ表示され、バックログと混在しない。

###### ＜カンバンボード表示仕様＞

- □□□ S-004-01 ボードは「プロダクトバックログ」「未着手」「進行中」「レビュー」「完了」の5カラムを横並びで表示する
- □□□ S-004-02 各カラムのヘッダーにカラム名とタスク件数バッジを表示する
- □□□ S-004-03 カラムが画面幅を超える場合、ボードを横スクロールできる
- □□□ S-004-04 タスク件数が多い場合、カラム内を縦スクロールできる
- □□□ S-004-05 タスクが0件のカラムには「タスクをここにドロップ」と案内テキストを表示する
- □□□ S-004-06 タスクカードに所属スプリント名を小さなチップで表示する
- □□□ S-004-07 プロダクトバックログカラムはスプリント未割り当て（sprintId が未設定）のタスクのみ表示する

### R-005 ステータス管理 #ステータス #ドラッグアンドドロップ

> **理由：** タスクの進行に合わせてステータスを更新できるようにするため

> **説明：** ドラッグ開始の閾値を5pxとし、クリック操作とドラッグ操作を明確に区別する。

###### ＜ステータス管理仕様＞

- □□□ S-005-01 タスクカードをドラッグ＆ドロップで別のカラムへ移動できる
- □□□ S-005-02 ドラッグ中は元の位置に半透明のゴースト表示を残す
- □□□ S-005-03 ドラッグ中にホバーしているカラムをハイライト表示する
- □□□ S-005-04 ドロップ先が有効なカラムでない場合、元の位置に戻す
- □□□ S-005-05 ドラッグ開始の閾値を5pxとし、誤動作を防ぐ

### R-006 優先順位管理 #優先順位 #並び替え

> **理由：** 同一ステータス内でタスクの優先順位を視覚的に表現できるようにするため

###### ＜優先順位管理仕様＞

- □□□ S-006-01 同一カラム内でタスクカードをドラッグ＆ドロップで上下に並び替えられる
- □□□ S-006-02 並び替えはドラッグ中にリアルタイムで反映する
- □□□ S-006-03 並び替えた順序はデータに保存され、次回起動時も維持される

### R-018 ステータス日本語化 #UI #日本語化

> **理由：** カンバンボードのステータスを日本語で表記し、ユーザーフレンドリーにするため

> **説明：** 既存のカラムID（todo、in-progress、done）はデータベース・APIの互換性維持のため変更しない。表示名のみを日本語化する。

###### ＜ステータス日本語化仕様＞

- □□□ S-018-01 カンバンボードのステータスカラムを日本語で表示する
- □□□ S-018-02 TODO → 未着手、In Progress → 進行中、Done → 完了に変更する
- □□□ S-018-03 既存のカラムID（todo、in-progress、done）は変更しない
- □□□ S-018-04 すべてのビュー（ボード・統計・スプリント）で統一した日本語表記を使用する

## ＜スプリント・進捗管理＞

### R-009 進捗表示 #進捗 #ヘッダー

> **理由：** スプリント全体の完了状況をヘッダーで常時把握できるようにするため

> **説明：** 完了の定義は「完了」カラムに移動済みのタスクのみとする。他カラムのタスクは進行中・未完了として扱う。

###### ＜進捗表示仕様＞

- □□□ S-009-01 ヘッダーに完了タスク数 / 全タスク数をチップで表示する
- □□□ S-009-02 ヘッダーにプログレスバーと完了率（%）を表示する
- □□□ S-009-03 進捗表示はタスクの追加・移動・削除に応じてリアルタイムで更新する
- □□□ S-009-04 「完了」カラムのタスクのみを完了としてカウントする

### R-011 スプリント管理 #スプリント #管理

> **理由：** タスクをスプリント単位でグルーピングし、イテレーションごとの計画・追跡を可能にするため

> **説明：** ログイン直後またはプロジェクト選択直後はアクティブなスプリントを自動選択する。アクティブなスプリントがない場合は「すべて」を選択する。プロダクトバックログカラムへのドラッグでスプリント割り当てを解除し（`sprintId = null`）、スプリントフィルタ中にバックログから他カラムへドラッグすると対象スプリントに自動割り当てする。

###### ＜スプリント管理仕様＞

- □□□ S-011-01 スプリントはスプリント名・開始日・終了日を持つ
- □□□ S-011-02 スプリントを新規作成・編集・削除できる
- □□□ S-011-03 各タスクをいずれかのスプリントに割り当てられる（任意）
- □□□ S-011-04 アクティブなスプリントを1つ選択でき、ボードはそのスプリントのタスクを表示する
- □□□ S-011-05 スプリントを「完了」にすると、未完了タスクを次スプリントまたはプロダクトバックログへ移動できる
- □□□ S-011-06 スプリント一覧画面でこれまでのスプリント履歴を確認できる
- □□□ S-011-07 スプリントデータは localStorage に永続化する
- □□□ S-011-08 プロダクトバックログカラムへドラッグしたタスクはスプリント割り当てを解除する
- □□□ S-011-09 スプリントフィルタ中にプロダクトバックログから他カラムへドラッグしたタスクは、対象スプリントに自動割り当てする
- □□□ S-011-10 ログイン直後またはプロジェクト選択直後はアクティブなスプリントが自動選択される。アクティブなスプリントがない場合は「すべて」を選択する

### R-013 ベロシティ #ベロシティ #スプリント

> **理由：** スプリントごとの完了量を定量化し、今後のスプリント計画の参考にできるようにするため

> **説明：** ストーリーポイントが未設定のタスクはベロシティ計算から除外し、除外件数を明示する。ベロシティは「完了」カラムに移動したタスクのストーリーポイント合計として算出する。

###### ＜ベロシティ仕様＞

- □□□ S-013-01 完了したスプリントのベロシティ（「完了」カラムに移動したタスクのストーリーポイント合計）を記録する
- □□□ S-013-02 スプリント一覧にベロシティを表示する
- □□□ S-013-03 過去スプリントのベロシティを棒グラフで可視化する
- □□□ S-013-04 直近スプリントの平均ベロシティを表示する
- □□□ S-013-05 ストーリーポイントが未設定のタスクはベロシティ計算から除外し、その件数を明示する

### R-014 バーンダウンチャート #バーンダウン #チャート

> **理由：** スプリント期間中の進捗を日次で可視化し、遅延リスクを早期に発見できるようにするため

> **説明：** 理想線はスプリント開始時の総ポイントから0に向かう直線。実績線が理想線を大きく上回る場合（遅延）に警告を表示する。

###### ＜バーンダウンチャート仕様＞

- □□□ S-014-01 アクティブなスプリントのバーンダウンチャートをボード上で確認できる
- □□□ S-014-02 チャートの横軸はスプリント期間（日付）、縦軸は残ストーリーポイントとする
- □□□ S-014-03 スプリント開始時の総ポイントから0に向かう理想線（直線）を表示する
- □□□ S-014-04 タスクが「完了」カラムに移動した日時を記録し、実績線を描画する
- □□□ S-014-05 実績線が理想線を大きく上回る（遅延している）場合に警告を表示する
- □□□ S-014-06 完了済みスプリントのバーンダウンチャートを履歴として参照できる

## ＜システム管理＞

### R-010 データ永続化 #データ #Supabase

> **理由：** デバイスやブラウザをまたいでデータにアクセスできるようにするため

> **説明：** 初回起動時に Supabase が空の場合は localStorage のデータを自動移行する（既存データを失わずクラウド移行するため）。

###### ＜データ永続化仕様＞

- □□□ S-010-01 タスク・スプリントデータは Supabase（クラウドDB）に保存する
- □□□ S-010-02 タスクの追加・編集・削除・並び替え・移動は即時 Supabase に反映する
- □□□ S-010-03 初回起動時（Supabase が空の場合）は localStorage のデータを自動移行する
- □□□ S-010-04 Supabase Realtime でデータ変更をリアルタイム受信する

### R-015 プロジェクト管理 #プロジェクト #管理

> **理由：** 用途や目的ごとにタスク・スプリントを独立して運用できるようにするため

> **説明：** プロジェクトが1つのみの場合は削除不可（最低1プロジェクトを常に保証）。プロジェクト削除時は紐づくタスク・スプリントもすべてカスケード削除される。

###### ＜プロジェクト管理仕様＞

- □□□ S-015-01 プロジェクトはヘッダーのドロップダウンセレクターで切り替える
- □□□ S-015-02 プロジェクトを新規作成・名前変更・削除できる
- □□□ S-015-03 プロジェクトを削除すると、紐づくタスク・スプリントもすべて削除される
- □□□ S-015-04 プロジェクトが1つしかない場合は削除できない
- □□□ S-015-05 最後に選択したプロジェクトを記憶し、次回アクセス時に自動で選択する
- □□□ S-015-06 異なるプロジェクトのタスク・スプリントは完全に分離して管理する
- □□□ S-015-07 Supabase の `projects` テーブルでプロジェクトを管理し、タスク・スプリントは `board_id` で紐づける

### R-017 ログイン認証 #認証 #セキュリティ

> **理由：** 不特定多数がアクセスできる公開URLにおいて、意図しない第三者の閲覧・操作を防ぐため

> **説明：** 認証情報はクライアントバンドルに含まれるため、機密性の高い用途には使用しない。環境変数（`VITE_AUTH_USER` / `VITE_AUTH_PASS`）が未設定の場合は認証をスキップする（ローカル開発時の利便性確保）。

###### ＜ログイン認証仕様＞

- □□□ S-017-01 環境変数（`VITE_AUTH_USER` / `VITE_AUTH_PASS`）にユーザーIDとパスワードを設定することで認証を有効化する
- □□□ S-017-02 環境変数が未設定の場合は認証をスキップしてボードを表示する
- □□□ S-017-03 未認証の場合はログイン画面を表示し、ボードへのアクセスをブロックする
- □□□ S-017-04 正しいユーザーID・パスワードを入力するとボードを表示する
- □□□ S-017-05 認証状態は `sessionStorage` に保持し、タブを閉じるまで有効とする
- □□□ S-017-06 認証情報はクライアントバンドルに含まれるため、機密性の高い用途には使用しない

### R-019 担当者マスタ管理 #担当者 #管理

> **理由：** プロジェクトごとに担当者を登録・管理し、タスク追加時の入力を効率化するため

> **説明：** 担当者を削除すると、その担当者が設定されているタスクの担当者は未設定になる（孤立データを防ぐ）。担当者が未登録のプロジェクトでは担当者欄を自由入力テキストフィールドのままにし、後方互換性を維持する。

###### ＜担当者マスタ管理仕様＞

- □□□ S-019-01 プロジェクトセレクターのドロップダウンから「👥 担当者を管理」モーダルを開ける
- □□□ S-019-02 担当者の追加・名前変更・削除ができる
- □□□ S-019-03 担当者を削除すると、その担当者が設定されているタスクの担当者は未設定になる
- □□□ S-019-04 プロジェクトごとにデフォルト担当者を1名設定できる
- □□□ S-019-05 「未設定」をデフォルトに戻す（デフォルトなし）設定ができる
