---

## 1. システム概要

スクラム開発におけるタスクを視覚的に管理するカンバンボードアプリケーション。  
チームメンバーはタスクの作成・編集・ステータス変更・優先度設定・担当者割り当てを行い、  
スプリントの進捗をリアルタイムで把握できる。  
データは Supabase（クラウドDB）に保存され、どのデバイスからでもアクセス可能。  
プロジェクト単位でタスク・スプリントを分離し、ヘッダーのセレクターで切り替えて運用できる。  
固定ユーザーID・パスワードによるログイン認証でアクセスを制限できる。  
プロジェクトごとに担当者リストを管理し、タスク追加時のデフォルト担当者を設定できる。

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| タスク | 作業単位。タイトル・説明・優先度・担当者・ストーリーポイントを持つ |
| カラム | タスクのステータスを表す列。プロダクトバックログ / 未着手 / 進行中 / レビュー / 完了 の5種 |
| ボード | 全カラムを並べた作業画面 |
| 優先度 | タスクの重要度。高・中・低 の3段階 |
| 担当者 | タスクを担当するメンバーの名前 |
| プロジェクト | タスク・スプリントをまとめる管理単位。名前・IDを持ち、Supabase の `projects` テーブルで管理する |
| スプリント | 一定期間（通常1〜4週間）の開発イテレーション。開始日・終了日・名称を持つ |
| ストーリーポイント | タスクの相対的な作業量見積もり値。整数で表す |
| ベロシティ | スプリントで完了したタスクのストーリーポイント合計値 |
| バーンダウンチャート | スプリント期間中の残ストーリーポイントの推移を示すグラフ |

## 3. 要求仕様

---

### R-001　タスク作成

> **理由：** 作業項目をシステムに登録できなければボードとして機能しないため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-001-01 | 各カラムの下部に「+ タスクを追加」ボタンを表示する | どのステータスからでも直接登録できるようにするため |
| S-001-02 | タスク追加フォームはタイトル（必須）・説明（任意）・優先度・担当者の入力欄を持つ | タスクに必要な情報を一度に登録できるようにするため |
| S-001-03 | タイトルが空の場合、追加ボタンを非活性にする | 識別不能なタスクの登録を防ぐため |
| S-001-04 | 追加完了またはキャンセル後、フォームを閉じてボタン表示に戻す | 画面領域を節約し、通常の操作に戻すため |
| S-001-05 | 優先度の初期値は「中」とする | 多くのタスクが中優先度であり、変更操作を省略できるようにするため |
| S-001-06 | タスク追加フォームにストーリーポイントの入力欄（任意・整数）を持つ | 作成時に見積もりを設定できるようにするため |

---

### R-002　タスク編集

> **理由：** 登録後に内容の変更が生じた場合に対応できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-002-01 | カード右上の「⋯」ボタンをクリックするとモーダルを表示する | カード面積を圧迫せず編集機能を提供するため |
| S-002-02 | モーダルではタイトル・説明・優先度・担当者をすべて編集できる | 登録時と同じ項目を変更できるようにするため |
| S-002-03 | タイトルが空の場合、保存ボタンを非活性にする | S-001-03 と同様の理由 |
| S-002-04 | Escキー押下またはモーダル背景クリックでモーダルを閉じる | キーボード操作・直感的な操作を可能にするため |
| S-002-05 | モーダルを開いた際、タイトル入力欄にフォーカスを当てる | 編集開始の手数を減らすため |
| S-002-06 | モーダルでストーリーポイントを編集できる | S-001-06 で設定した値を後から変更できるようにするため |

---

### R-003　タスク削除

> **理由：** 不要になったタスクをボードから除去できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-003-01 | 削除はタスク編集モーダル内の「削除」ボタンから行う | 誤操作を防ぐためカード上に直接削除ボタンを設けない |
| S-003-02 | 削除ボタン押下時は確認メッセージを表示し、再度確認を求める | 誤削除を防ぐための二段階確認とするため |
| S-003-03 | 確認後に削除を実行し、モーダルを閉じる | 削除完了をユーザーが視覚的に確認できるようにするため |

---

### R-004　カンバンボード表示

> **理由：** タスクの全体像とステータス分布を一覧で把握できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-004-01 | ボードは「プロダクトバックログ」「未着手」「進行中」「レビュー」「完了」の5カラムを横並びで表示する | スクラム開発の一般的なワークフローを表現するため |
| S-004-02 | 各カラムのヘッダーにカラム名とタスク件数バッジを表示する | 各ステータスのタスク量を即座に把握できるようにするため |
| S-004-03 | カラムが画面幅を超える場合、ボードを横スクロールできる | 全カラムを表示しつつ小さい画面にも対応するため |
| S-004-04 | タスク件数が多い場合、カラム内を縦スクロールできる | カラム高さを固定してボード全体の視認性を保つため |
| S-004-05 | タスクが0件のカラムには「タスクをここにドロップ」と案内テキストを表示する | ドロップ可能な領域であることを示すため |
| S-004-06 | タスクカードに所属スプリント名を小さなチップで表示する | ボードを見ながらスプリントの割り当てを確認できるようにするため |
| S-004-07 | プロダクトバックログカラムはスプリント未割り当て（sprintId が未設定）のタスクのみ表示する | スプリント管理中にバックログと進行中スプリントのタスクが混在しないようにするため |

---

### R-005　ステータス管理（カラム間移動）

> **理由：** タスクの進行に合わせてステータスを更新できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-005-01 | タスクカードをドラッグ＆ドロップで別のカラムへ移動できる | マウス操作で直感的にステータスを変更できるようにするため |
| S-005-02 | ドラッグ中は元の位置に半透明のゴースト表示を残す | 移動元の位置を把握しやすくするため |
| S-005-03 | ドラッグ中にホバーしているカラムをハイライト表示する | ドロップ先を視覚的に確認できるようにするため |
| S-005-04 | ドロップ先が有効なカラムでない場合、元の位置に戻す | 不正な操作を防ぐため |
| S-005-05 | ドラッグ開始の閾値を5pxとし、誤動作を防ぐ | クリックとドラッグを明確に区別するため |

---

### R-006　優先順位管理（カラム内並び替え）

> **理由：** 同一ステータス内でタスクの優先順位を視覚的に表現できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-006-01 | 同一カラム内でタスクカードをドラッグ＆ドロップで上下に並び替えられる | 優先度の高いタスクを上に配置して順序を管理できるようにするため |
| S-006-02 | 並び替えはドラッグ中にリアルタイムで反映する | 最終的な位置を確認しながら操作できるようにするため |
| S-006-03 | 並び替えた順序はデータに保存され、次回起動時も維持される | 操作結果が失われないようにするため |

---

### R-007　優先度表示

> **理由：** タスクの重要度を一目で把握し、対応順序の判断を支援するため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-007-01 | タスクカードに優先度バッジ（高・中・低）を表示する | カードの一覧状態で優先度を確認できるようにするため |
| S-007-02 | 優先度ごとに色を変える（高：赤、中：黄、低：緑） | 色による即時識別を可能にするため |

---

### R-008　担当者管理

> **理由：** 誰がどのタスクを担当しているかをボード上で把握できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-008-01 | タスクに担当者名を設定できる（任意項目） | 担当者がいないタスクも管理できるようにするため |
| S-008-02 | 担当者が設定されている場合、カード下部に名前の頭文字アバターと名前を表示する | 視覚的に担当者を識別しやすくするため |
| S-008-03 | アバターの背景色は担当者名から一意に決定する | 同じ担当者を常に同じ色で識別できるようにするため |

---

### R-009　進捗表示

> **理由：** スプリント全体の完了状況をヘッダーで常時把握できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-009-01 | ヘッダーに完了タスク数 / 全タスク数をチップで表示する | 絶対数で進捗を把握できるようにするため |
| S-009-02 | ヘッダーにプログレスバーと完了率（%）を表示する | 全体の進捗を視覚的・数値的に把握できるようにするため |
| S-009-03 | 進捗表示はタスクの追加・移動・削除に応じてリアルタイムで更新する | 常に最新の状態を反映するため |
| S-009-04 | 「完了」カラムのタスクのみを完了としてカウントする | 完了の定義を明確にするため |

---

### R-010　データ永続化

> **理由：** デバイスやブラウザをまたいでデータにアクセスできるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-010-01 | タスク・スプリントデータは Supabase（クラウドDB）に保存する | どのデバイスからでもデータにアクセスできるようにするため |
| S-010-02 | タスクの追加・編集・削除・並び替え・移動は即時 Supabase に反映する | 操作後すぐに他デバイスでも同じ状態が見えるようにするため |
| S-010-03 | 初回起動時（Supabase が空の場合）は localStorage のデータを自動移行する | 既存データを失わずにクラウド移行できるようにするため |
| S-010-04 | Supabase Realtime でデータ変更をリアルタイム受信する | 複数デバイスや複数ユーザーの変更を即時反映するため |

---

### R-011　スプリント管理

> **理由：** タスクをスプリント単位でグルーピングし、イテレーションごとの計画・追跡を可能にするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-011-01 | スプリントはスプリント名・開始日・終了日を持つ | スプリントを識別し、期間を明確にするため |
| S-011-02 | スプリントを新規作成・編集・削除できる | スプリント計画の変更に対応するため |
| S-011-03 | 各タスクをいずれかのスプリントに割り当てられる（任意） | スプリントに含めないプロダクトバックログのタスクも扱えるようにするため |
| S-011-04 | アクティブなスプリントを1つ選択でき、ボードはそのスプリントのタスクを表示する | 現在進行中のスプリントに集中して作業できるようにするため |
| S-011-05 | スプリントを「完了」にすると、未完了タスクを次スプリントまたはプロダクトバックログへ移動できる | スプリント終了の手続きを定義するため |
| S-011-06 | スプリント一覧画面でこれまでのスプリント履歴を確認できる | 過去のスプリントデータを参照できるようにするため |
| S-011-07 | スプリントデータは localStorage に永続化する | R-010 と同様の理由 |
| S-011-08 | プロダクトバックログカラムへドラッグしたタスクはスプリント割り当てを解除する | プロダクトバックログの定義（未割り当て）と操作を一致させるため |
| S-011-09 | スプリントフィルタ中にプロダクトバックログから他カラムへドラッグしたタスクは、対象スプリントに自動割り当てする | 手動でスプリント設定する手間を省くため |
| S-011-10 | ログイン直後またはプロジェクト選択直後はアクティブなスプリントが自動選択される。アクティブなスプリントがない場合は「すべて」を選択する | 毎回手動で切り替える手間を省くため |

---

### R-012　ストーリーポイント

> **理由：** タスクの作業量を数値化し、スプリント計画とベロシティ計測の基礎データとするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-012-01 | タスクにストーリーポイント（正の整数）を設定できる（任意） | 見積もりが確定していないタスクも扱えるようにするため |
| S-012-02 | ストーリーポイントが設定されたカードには値をバッジ表示する | カード一覧で作業量を確認できるようにするため |
| S-012-03 | 各カラムのヘッダーにそのカラム内タスクのストーリーポイント合計を表示する | ステータスごとの作業量を把握できるようにするため |
| S-012-04 | スプリントのストーリーポイント合計（計画値）をスプリント情報として表示する | スプリントの規模感を把握できるようにするため |
| S-012-05 | カード上のストーリーポイントバッジをクリックすると数値を直接編集できる | モーダルを開かずにポイントを素早く修正できるようにするため |
| S-012-06 | ストーリーポイント未設定のカードにも「– pt」の編集可能バッジを表示する | 未設定に気づき、そのまま設定操作に入れるようにするため |

---

### R-013　ベロシティ

> **理由：** スプリントごとの完了量を定量化し、今後のスプリント計画の参考にできるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-013-01 | 完了したスプリントのベロシティ（「完了」カラムに移動したタスクのストーリーポイント合計）を記録する | スプリント終了時の実績値を保持するため |
| S-013-02 | スプリント一覧にベロシティを表示する | スプリントごとの実績を一覧で比較できるようにするため |
| S-013-03 | 過去スプリントのベロシティを棒グラフで可視化する | 推移のトレンドを視覚的に把握できるようにするため |
| S-013-04 | 直近スプリントの平均ベロシティを表示する | 次スプリントの計画ポイント数の目安として使えるようにするため |
| S-013-05 | ストーリーポイントが未設定のタスクはベロシティ計算から除外し、その件数を明示する | 計算結果の意味を正確に伝えるため |

---

### R-014　バーンダウンチャート

> **理由：** スプリント期間中の進捗を日次で可視化し、遅延リスクを早期に発見できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-014-01 | アクティブなスプリントのバーンダウンチャートをボード上で確認できる | 現在の進捗状況をリアルタイムで把握できるようにするため |
| S-014-02 | チャートの横軸はスプリント期間（日付）、縦軸は残ストーリーポイントとする | バーンダウンの標準的な表現形式に従うため |
| S-014-03 | スプリント開始時の総ポイントから0に向かう理想線（直線）を表示する | 実績と計画のズレを視覚的に比較できるようにするため |
| S-014-04 | タスクが「完了」カラムに移動した日時を記録し、実績線を描画する | 実際の消化ペースを正確に反映するため |
| S-014-05 | 実績線が理想線を大きく上回る（遅延している）場合に警告を表示する | 遅延を早期に認識できるようにするため |
| S-014-06 | 完了済みスプリントのバーンダウンチャートを履歴として参照できる | 振り返り（レトロスペクティブ）の資料として活用できるようにするため |

### R-015　プロジェクト管理

> **理由：** 用途や目的ごとにタスク・スプリントを独立して運用できるようにするため（例：自分用・デモ用）

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-015-01 | プロジェクトはヘッダーのドロップダウンセレクターで切り替える | URLを変えずに単一URLでプロジェクトを切り替えられるようにするため |
| S-015-02 | プロジェクトを新規作成・名前変更・削除できる | 用途に合わせてプロジェクトを柔軟に管理できるようにするため |
| S-015-03 | プロジェクトを削除すると、紐づくタスク・スプリントもすべて削除される | 孤立したデータが残らないようにするため |
| S-015-04 | プロジェクトが1つしかない場合は削除できない | 最低1つのプロジェクトが常に存在することを保証するため |
| S-015-05 | 最後に選択したプロジェクトを記憶し、次回アクセス時に自動で選択する | 毎回プロジェクトを選び直す手間を省くため |
| S-015-06 | 異なるプロジェクトのタスク・スプリントは完全に分離して管理する | 複数の用途を干渉なく並行運用できるようにするため |
| S-015-07 | Supabase の `projects` テーブルでプロジェクトを管理し、タスク・スプリントは `board_id` で紐づける | データベースレベルでプロジェクトを分離するため |

---

### R-016　タスク期限管理

> **理由：** タスクごとに期限を設定し、遅延リスクを視覚的に把握できるようにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-016-01 | タスクカードに期限バッジ（📅 MM/DD 形式）を表示する | 一覧から期限を一目で確認できるようにするため |
| S-016-02 | 期限バッジをクリックすると日付入力フィールドに切り替わりインライン編集できる | ボードを離れずに素早く期限を設定・変更できるようにするため |
| S-016-03 | 期限超過のタスクは赤色、2日以内は橙色、設定済みは紫色でバッジを表示する | 緊急度を色で直感的に把握できるようにするため |
| S-016-04 | 完了カラムのタスクは期限超過でも色変化しない | 完了済みタスクの視覚的ノイズを防ぐため |
| S-016-05 | タスク編集モーダルからも期限を設定・変更できる | 詳細編集時にもまとめて設定できるようにするため |
| S-016-06 | 期限は Supabase の `due_date` カラム（text 型、YYYY-MM-DD）に保存する | クロスプラットフォームで日付を一貫して扱えるようにするため |

### R-017　ログイン認証

> **理由：** 不特定多数がアクセスできる公開URLにおいて、意図しない第三者の閲覧・操作を防ぐため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-017-01 | 環境変数（`VITE_AUTH_USER` / `VITE_AUTH_PASS`）にユーザーIDとパスワードを設定することで認証を有効化する | コードを変更せずに認証情報を管理できるようにするため |
| S-017-02 | 環境変数が未設定の場合は認証をスキップしてボードを表示する | ローカル開発時に認証なしで動作確認できるようにするため |
| S-017-03 | 未認証の場合はログイン画面を表示し、ボードへのアクセスをブロックする | 認証前にデータが表示されないようにするため |
| S-017-04 | 正しいユーザーID・パスワードを入力するとボードを表示する | 認証後すぐにボードを操作できるようにするため |
| S-017-05 | 認証状態は `sessionStorage` に保持し、タブを閉じるまで有効とする | ページ移動・リロードのたびにログインし直す手間を省くため |
| S-017-06 | 認証情報はクライアントバンドルに含まれるため、機密性の高い用途には使用しない | セキュリティ上の制約をシステムとして明示するため |

### R-018　カンバンボードステータス日本語化

> **理由：** カンバンボードのステータスを日本語で表記し、ユーザーフレンドリーにするため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-018-01 | カンバンボードのステータスカラムを日本語で表示する | 日本ユーザーにとってわかりやすいUIを提供するため |
| S-018-02 | TODO → 未着手、In Progress → 進行中、Done → 完了に変更する | スクラム開発の標準的な日本語ステータス名を採用するため |
| S-018-03 | 既存のカラムID（todo、in-progress、done）は変更しない | データベースやAPI の互換性を保つため |
| S-018-04 | すべてのビュー（ボード・統計・スプリント）で統一した日本語表記を使用する | 全体的な一貫性を確保するため |

### R-019　担当者管理

> **理由：** プロジェクトごとに担当者を登録・管理し、タスク追加時の入力を効率化するため

| 仕様ID | 仕様 | 理由 |
|--------|------|------|
| S-019-01 | プロジェクトセレクターのドロップダウンから「👥 担当者を管理」モーダルを開ける | プロジェクトに紐づく担当者を同じ文脈で管理できるようにするため |
| S-019-02 | 担当者の追加・名前変更・削除ができる | 人員変更に柔軟に対応できるようにするため |
| S-019-03 | 担当者を削除すると、その担当者が設定されているタスクの担当者は未設定になる | 存在しない担当者が残留してデータの整合性が崩れるのを防ぐため |
| S-019-04 | プロジェクトごとにデフォルト担当者を1名設定できる | タスク追加時の入力を省略できるようにするため |
| S-019-05 | 「未設定」をデフォルトに戻す（デフォルトなし）設定ができる | 担当者を指定しないワークフローにも対応するため |
| S-019-06 | 担当者が登録されている場合、タスク追加フォーム・編集モーダルの担当者欄をドロップダウンに切り替える | 入力ミスを防ぎ、一貫した担当者名を維持するため |
| S-019-07 | 担当者が未登録のプロジェクトでは、担当者欄を自由入力のテキストフィールドのままにする | 後方互換性を維持し、担当者管理なしでも使えるようにするため |
| S-019-08 | 担当者情報は Supabase の `assignees` テーブルにプロジェクトIDで紐づけて保存する | プロジェクト間でデータを分離するため |

## 4. 非機能要求

| ID | 分類 | 要求 |
|----|------|------|
| NF-001 | レスポンス | ドラッグ操作・カード追加はユーザー操作から 100ms 以内に画面反映する |
| NF-002 | 対応ブラウザ | Chrome / Edge / Firefox / Safari（iOS・Android 含む）の最新版で動作する |
| NF-003 | デプロイ | Vercel へ静的サイトとしてデプロイできる構成とする |
| NF-004 | チャート描画 | バーンダウンチャート・ベロシティグラフはデータ変更から 200ms 以内に再描画する |
| NF-005 | インフラ | データは Supabase（PostgreSQL）に保存し、Vercel（静的ホスティング）で公開する |
| NF-006 | リアルタイム | Supabase Realtime により、複数デバイスの変更を 1 秒以内に反映する |
| NF-007 | モバイル対応 | スマートフォンブラウザでレイアウトが崩れず、タッチ操作でドラッグ&ドロップが動作する |
